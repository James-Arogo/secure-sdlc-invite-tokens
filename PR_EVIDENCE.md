# PR Evidence: Secure Invite Tokens

This file captures the rubric-critical evidence for the secure SDLC trail. GitHub PR: [#1](https://github.com/James-Arogo/secure-sdlc-invite-tokens/pull/1) contains the same threat-model note and self-review checklist.

## Threat-Model Note

Feature: `secure_invites` creates and verifies onboarding invite tokens for a specific audience such as `partner-portal`.

Threats considered:

- Token tampering: an attacker modifies `sub`, `aud`, or `exp` after obtaining a token.
- Replay or expiry bypass: an attacker reuses an old token or extends the expiry timestamp.
- Audience confusion: a token issued for `partner-portal` is replayed against `admin-console`.
- Malformed claims: invalid payloads are submitted to bypass validation or trigger parser failure.

Controls implemented:

- HMAC-SHA256 signature over canonical JSON payloads.
- Constant-time signature verification with `hmac.compare_digest`.
- Explicit audience binding in `verify_invite_token`.
- Expiry validation and malformed/incomplete claim rejection.

Quoted threat-model line from `docs/prs/001-secure-invite-tokens.md`: "sign canonical JSON payloads with HMAC-SHA256" and "enforce explicit audience during verification".

## Vulnerability and Remediation

Vulnerable baseline commit: [`86751e3`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/commit/86751e3) added invite tokens as base64url JSON without integrity protection. Root cause: the verifier trusted decoded claims without proving they were issued by the server.

Review finding commit: [`1619f8b`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/commit/1619f8b) documented that an attacker could change `sub` or `exp`, re-encode the token, and submit it.

Remediation commit: [`42ec7fc`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/commit/42ec7fc) fixed the vulnerability.

One-line remediation diff summary: `secure_invites/tokens.py` changed from unsigned base64url JSON to `payload.signature`, signing canonical JSON with HMAC-SHA256 and rejecting invalid signatures with `hmac.compare_digest`.

## Code Review / Self-Review Checklist

Reviewer: self-review documented in `docs/reviews/001-security-review.md` and commit `1619f8b`.

Review comments captured:

- Baseline token format is only base64url-encoded JSON.
- Attacker can decode, modify `sub` or `exp`, re-encode, and submit the token.
- Do not ship until payload integrity is enforced.

Checklist from the remediation review:

- [x] Sign canonical payloads with HMAC-SHA256.
- [x] Verify signatures with constant-time comparison.
- [x] Reject tampered tokens in tests.
- [x] Keep expiry and audience enforcement tests.
- [x] Remove generated bytecode from git and add `.gitignore`.

## Security Tests

Test file: [`tests/test_invites.py`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/blob/main/tests/test_invites.py)

Security-relevant tests:

- `test_rejects_tampered_payload`: modifies the encoded payload while keeping the original signature and expects `InvalidToken` with a signature error.
- `test_rejects_expired_tokens`: verifies expired tokens are rejected.
- `test_rejects_wrong_audience`: verifies a token for `partner-portal` cannot be used for `admin-console`.
- `test_rejects_wrong_secret`: verifies a token cannot be validated with a different HMAC secret.
- `test_round_trips_claims`: verifies a valid signed token returns the intended subject, audience, issued-at, and expiry claims.

Validation command:

```bash
python3 -m unittest discover -s tests
```

Latest local validation result: 5 tests passed.
