# Submission Write-Up: Secure SDLC Invite Tokens

## 1. Feature Built and Threat Model

I built a small Python feature called `secure_invites` that creates and verifies invite tokens for an onboarding flow. The reason this feature matters from a security perspective is that invite tokens act like temporary authorization: whoever presents a valid token can enter a protected workflow for a specific audience, such as `partner-portal`.

The main threats I considered were token tampering, replay, expiry bypass, and audience confusion. An attacker who obtains a token should not be able to change the subject from `user-123` to an admin user, extend the expiry time, or reuse the same token against a different audience such as `admin-console`. The final implementation protects the token payload with HMAC-SHA256, validates expiry, validates the expected audience, rejects malformed claims, and uses `hmac.compare_digest` for constant-time signature comparison.

## 2. Vulnerability Found and Remediated

The initial implementation in commit [`86751e3`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/commit/86751e3) added invite tokens as base64url-encoded JSON. That was intentionally vulnerable because base64 is only encoding, not integrity protection. The root cause was that the verifier trusted decoded JSON claims without proving that the claims were issued by the server. An attacker could decode the payload, change fields like `sub` or `exp`, re-encode it, and submit the tampered token.

The review finding is documented in commit [`1619f8b`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/commit/1619f8b) and in `docs/reviews/001-security-review.md`. The remediation is commit [`42ec7fc`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/commit/42ec7fc). One-line diff summary: `secure_invites/tokens.py` changed from unsigned base64url JSON to a `payload.signature` format, signing canonical JSON with HMAC-SHA256 and rejecting invalid signatures with `hmac.compare_digest`. This fixed the claim-tampering vulnerability while keeping the existing expiry and audience checks.

## 3. Code Review Evidence

Code review is represented by `docs/reviews/001-security-review.md`, commit [`1619f8b`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/commit/1619f8b), and the GitHub PR created for this revision. The review explicitly blocked shipping because the baseline token format was unsigned and therefore tamperable. The reviewer comments were: the baseline token was only base64url-encoded JSON, an attacker could change `sub` or `exp`, and the feature should not ship until payload integrity was enforced.

The remediation checklist was: sign canonical payloads with HMAC-SHA256, verify signatures with constant-time comparison, reject tampered tokens in tests, keep expiry/audience tests, and remove generated bytecode from git. The follow-up remediation commit [`42ec7fc`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/commit/42ec7fc) addresses those review comments directly.

## 4. Security Tests Written

The security-relevant tests are in [`tests/test_invites.py`](https://github.com/James-Arogo/secure-sdlc-invite-tokens/blob/main/tests/test_invites.py). Specific tests are `test_rejects_tampered_payload`, `test_rejects_expired_tokens`, `test_rejects_wrong_audience`, `test_rejects_wrong_secret`, and `test_round_trips_claims`.

The most important regression test is `test_rejects_tampered_payload`. It creates a legitimate token, modifies the encoded payload to change the subject, keeps the original signature, and verifies that the implementation rejects it with a signature error. `test_rejects_expired_tokens` proves expired tokens fail. `test_rejects_wrong_audience` proves audience binding. `test_rejects_wrong_secret` proves the HMAC secret is required. The test suite can be run with:

```bash
python3 -m unittest discover -s tests
```

At submission time, the suite passed with 5 tests.
